Memory and pointers

Integers are indices

Integers have lots of useful properties. They’re closed over addition, subtraction, multiplication, and exponentiation, which means the result of applying any of those operations to a pair of integers yields an integer. This means you can do most kinds of arithmetic on integers without ever worrying about losing precision (as long as the result doesn’t cross an overflow boundary).

Javascript people! Yes, I know you use floats as array indices! Except they’re not really floats. Most Javascript engines store whole numbers as binary integers, if they are small enough to be representable by the integer type. Incidentally, that’s more or less the range of valid array indices in Javascript.

More importantly, you can use integers as indices, which means if you have an integer n, you can use it to refer to the nth thing in a sequence. That’s because integers have an inherent predecessor and successor. It makes sense to ask what comes after 22, but it wouldn’t make much sense to ask what comes after 22.319891 or any other fraction.

This property is important because it lets us talk about memory. Memory is made up of memory cells which computers use to store bits of data. Memory cells are grouped into bytes (usually containing 8 bits each) called memory locations. Memory locations are sequentially numbered giving each one a unique memory address. The address space, or number of memory locations you can address, is limited by the range of your pointers, integer indices representing memory addresses. An 8 bit pointer can address up to 256 memory locations.

0 1 2 3 252 253 254 255

Pointers are indices in memory

Pointers are numbers. Those numbers are memory addresses, indices of bytes in memory. Since that tends to confuse people, even seasoned programmers from languages that have no concept of pointers, it helps to go slow.

Here, we are binding all of the memory to the UInt8 data type so you have some way of telling what binary value each memory location holds. The only memory location that actually has to be an unsigned integer is the pointer itself. All the other locations could be any type.

Here, we have a pointer in memory location 59. The 8 bit unsigned integer value of the pointer is 63. That number 63 is the index of the memory location that contains the pointer’s pointee, 17.

58 59 60 61 62 63 64 65
7 63 0 0 8 17 52 0

Don’t confuse the address of the pointee with the pointer’s own address. The address of the pointee is the value of the pointer, it has nothing to do with where in memory the pointer itself lives. In fact, you can have the pointer point to itself. Here, the value of memory location 59 is the number 59.

58 59 60 61 62 63 64 65
7 59 0 0 8 17 52 0

Similarly, don’t confuse the value of the pointee with the value of the pointer (the address of the pointee). You can swap out the contents of the memory location the pointer points to without ever changing the pointer.

58 59 60 61 62 63 64 65
7 63 0 0 8 255 52 0
58 59 60 61 62 63 64 65
7 63 0 0 8 0 52 0
58 59 60 61 62 63 64 65
7 63 0 0 8 69 52 0

Changing the value of a pointer changes which memory location it refers to. You can increment a pointer to iterate through a sequence of memory locations.

58 59 60 61 62 63 64 65
7 63 0 0 8 17 52 0
58 59 60 61 62 63 64 65
7 64 0 0 8 0 52 0
58 59 60 61 62 63 64 65
7 65 0 0 8 69 52 0

Pointers have ordering and arithmetic operations

Warning!

Not all languages define pointer comparisons and pointer arithmetic the same way Swift does, if at all.

The value of a pointer is always interpreted as an unsigned integer. You can perform addition or subtraction — pointer arithmetic — on a pointer changing which memory location it points to. You can also compare pointer values. A pointer with a smaller unsigned integer value is said to reference an earlier memory location than a pointer with a larger unsigned integer value. This means an 8 bit pointer with the bit pattern 1111,1111 compares greater than a pointer with the bit pattern 0000,0001. (Even though printing the first pointer as a signed integer would yield “−1”.)

Since they walk and talk like unsigned integers, we could use the data type UInt8 for all of our 8 bit pointers. However, since pointers are conceptually indices of memory locations, not numeric quantities, it makes sense for us to define a new data type just for pointers: UnsafeRawPointer.

One of the reasons pointers are called “Unsafe” is that they can easily be set to reference a memory location that’s inaccessible, or nonexistent. For example, if you only have memory locations numbered 0 through 120, the pointer 121 points to a memory location that doesn’t exist.

The bigger reason is that they can create inconsistencies in the type system, and potentially undermine type safety if you’re not careful. To understand why, you need to first understand how Swift’s static typing system works.

Variables carry type information

Static typing is not the same as strong typing, although the two concepts are correlated. C/C++’s type system, for example, is both weak and mostly static.

In dynamically typed languages, type information is carried by the data values themselves. This means, for example, that each Int8 value would have an extra piece of data attached to it labeling it as an Int8. Normally this piece of data is a pointer to a table of data types. Moving the object around requires moving both pieces of data together as a unit. In contrast, in statically typed languages, type information is carried by variables. Swift’s type system is mostly static.

Some scripting languages like Python instead implement variables using hash maps, a slower but more flexible approach.

Variables are human-readable names we give to memory locations when coding in a high level language like Swift. In most languages, variables are implemented as hardcoded pointer offsets in the machine code the compiler generates. The offsets are relative to a hidden pointer called the frame pointer that lives in a register “outside” of memory, and is not directly accessible in Swift.

From now on, you’ll start seeing raw binary values and memory addresses written in hexadecimal (base 16) notation instead of 0s and 1s. Each hexadecimal digit represents four bits. In this book, hexadecimal values will always be prefixed with the letter x.

Variables carry type information that tells us how to interpret the data in the memory. It’s important to understand that in this system, the type is seen only by the author and the compiler; it’s not represented or stored anywhere in memory. Here, the variable x is declared as a UInt8, and the variable y is declared as an Int8. As notation, we write the variable name and variable type separated by a colon with the type to the right, like this: x:UInt8, y:Int8.

The type of a variable does not need to be a concrete type. It’s possible for the data value to carry a concrete type that the variable doesn’t know about. Such a data value is called a polymorphic object. All of the data types in this chapter are concrete.

Because Swift is a strongly typed language, once a variable binds a type to data, it is impossible to make the variable bind a different type later in the program.

Raw pointers point to arbitrary memory

If you only ever access memory through variables, then Swift can enforce type safety perfectly since the compiler tracks type consistency through variables. However, pointers provide a way to reference and retrieve data values that live in memory locations not attached to a variable.

x y ptr
x3A x3B x3C x3D x3E x3F x40 x41
xDC 130 −5 x3B
Frame pointer x3E
Variable Offset Raw value Typed value
x:UInt8 +0 x82 130
y:Int8 +1 xFB −5
ptr:UnsafeRawPointer +2 x3B x3B

The type of the variable ptr is UnsafeRawPointer, which doesn’t tell you anything about the type of its pointee. The pointee, which lives in memory location x3B and has the value xDC, is not (necessarily) an UnsafeRawPointer.

Typed pointers specify the pointee’s type

Like almost all languages that support pointers, Swift provides typed pointer types (yes, you read that right) that specify the pointee’s type. This is as opposed to a pointer type that doesn’t specify the pointee’s type, which is called a raw pointer, hence the name UnsafeRawPointer. In Swift, we use the UnsafePointer type to represent a typed pointer, and put the pointee type in angle brackets like this: UnsafePointer<Int8>.

Remember, the pointee type only applies to the pointee — the −36 living in memory location x3B. The pointer value x3B living in location x40 is still a pointer.
x y ptr
x3A x3B x3C x3D x3E x3F x40 x41
−36 130 −5 x3B
Frame pointer x3E
Variable Offset Raw value Typed value
x:UInt8 +0 x82 130
y:Int8 +1 xFB −5
ptr:UnsafePointer<Int8> +2 x3B x3B
Swift generics are something of a mixture between C# generics, and Rust generics and C++ templates.

When the pointee type is combined with UnsafePointer like that, it becomes a type parameter or a generic parameter. (The two terms mean the same thing.) A type that takes a generic parameter is called a generic type. If we just want to talk about the generic type in general without mentioning a specific type parameter, we usually write a letter like “T” or “U” where the type parameter goes, or use a placeholder word describing what “kind” of type belongs there, for example: UnsafePointer<Pointee>.

Why not just define new data types like “UnsafePointerToInt8”, “UnsafePointerToUInt8”, “UnsafePointerToUnsafeRawPointer”, etc, instead of using a type parameter? Type parameters give you flexibility that providing a bunch of standard library pointer types can’t. For example, you can make UnsafePointer<Pointee>s to custom types you define, and make UnsafePointer<Pointee>s to other UnsafePointer<Pointee>s.

Typed pointers can still create inconsistencies

There’s a reason UnsafePointer<Pointee> still has the “Unsafe” prefix. While typed pointers are safer than raw pointers, they are still unsafe. Because pointers point into arbitrary memory, there is no way of guaranteeing the values there are actually valid for that type. It can point to junk values, or to something that’s meant to be interpreted as a different type than the given pointee type. In the example below, ptr2:UnsafePointer<UInt8> points to the same location as ptr1:UnsafePointer<Int8>, except ptr2 is viewing the value as a UInt8. So while ptr1 sees the xDC value as a −36, ptr2 sees it as a 220.

x y ptr1 ptr2
x3A x3B x3C x3D x3E x3F x40 x41
xDC 130 −5 x3B x3B
Frame pointer x3E
Variable Offset Raw value Typed value
x:UInt8 +0 x82 130
y:Int8 +1 xFB −5
ptr1:UnsafePointer<Int8> +2 x3B x3B
ptr2:UnsafePointer<UInt8> +3 x3B x3B

There’s lots of cases where you want to do something like this. However, Swift’s type safety won’t protect you from mismatch bugs here.

Multibyte values

So far all of our data types have been exactly 8 bits long. You can’t count past 255 (unsigned), or 127 (signed) without overflow. The more pressing problem is that since our pointers are 8 bits long, we can only address 255 memory locations. That’s not a lot of memory. To address more memory, you need longer integer types, and to get longer integer types, you need more bits.

One way to get more bits is to make the bytes longer, say 16 bits instead of 8. The LC-3 does this. The downside of this is now, 16 bits is also your minimum integer length, since it’s fairly hard to extract just part of a byte.

16 bit and 64 bit integers are equivalent to short and long in C-family languages.

Instead, every “real” architecture strings multiple bytes together to get longer integers. Two bytes make a 16 bit integer, four make a 32 bit integer, and eight make a 64 bit integer. In Swift, these types are called Int16, Int32, and Int64. Their unsigned varieties are called UInt16, UInt32, and UInt64.

The address of a multibyte value is the address of its first byte (the byte with the lowest address).

The order that the bytes are sequenced within each multibyte value is called its endianness or byte order. A big-endian integer stores its most significant bits first.

(big-end.)

A little-endian integer stores its least significant bits first. This means rebinding a little-endian integer to the UInt8 type will read its least significant 8 bits, not its most significant 8 bits.

(little-end.)

Byte order does not affect the order of the bits inside each byte. The bits in the first byte of a little-endian integer are exactly the same as the bits in the last byte of a big-endian integer.

Little-endian is more common than big-endian. In general, you should think of multibyte values as atomic, i.e. as a single, opaque chunk. Never try to split or splice a multibyte integer without knowing the byte order of the platform.

Memory alignment

Almost every modern processor can retrieve multiple bytes of memory at a time. Otherwise it would take forever to get all eight bytes out of an Int64. However, they can’t just grab any two or four or eight bytes. They can only access slices of memory that are situated on multiples of two, four, eight etc.

1-byte access
x28 x29 x2A x2B x2C x2D x2E x2F
2-byte access
x28 x29 x2A x2B x2C x2D x2E x2F
4-byte access
x28 x29 x2A x2B x2C x2D x2E x2F
8-byte access
x28 x29 x2A x2B x2C x2D x2E x2F

Failing to align a multibyte integer with a memory slice can have a negative impact on performance, since the integer will be split across two slices, and so the CPU will have to retrieve two slices instead of one, as well as perform the appropriate shifts and fuse the integer back together in a register.

Swift will automatically align all multibyte values to the correct memory alignment. However, it’s important to be aware of this because it can cause empty spaces called padding bytes to appear between different-sized values. For example, a padding byte appears between these three integers below.

x28 x29 x2A x2B x2C x2D x2E x2F
Int16 Int8 Int32

This becomes important when we discuss structures and arrays.

The pointer size is the default integer size

Since Swift has four different sizes of signed integers, and four different sizes of unsigned integers, it can get confusing deciding which integer type to use when, and making them all play nice with one another. It would be fantastic to just settle on a single “default” integer type that you could use for everything, and leave the zoo of sized integers for specialized uses.

In C and C++, most people picked Int32 for this, mainly because the extra bit of range in an unsigned integer wasn’t worth the inability to represent negative numbers, and because all of the other sizes had longer names that required more typing. (Int32 was just written as int in C; to get the other sizes, you had to type short or long before it.)

Don’t infer from this that Swift’s pointers are signed. Pointers have no concept of sign, and if you remember from the last chapter, signed representation is irrelevant to the mechanics of arithmetic itself; it’s merely a way to interpret the result.

In Swift, the default integer size is signed, and is the same size as a pointer. This makes it easier to do pointer arithmetic, since you don’t have to convert types to make the two operands compatible. Making integers and pointers the same size (and therefore having the same alignment) also makes them pack better, wasting fewer padding bytes.

In the past, pointers were 32 bits long. Machines that operated with 32 bit pointers were “32 bit machines”. Since a UInt32 has a maximum value of 4,294,967,295, this meant you could only address about 4 gigabytes of RAM. That used to be plenty, but today, computers regularly feature upwards of 16 gigabytes of RAM, necessitating the transition to 64 bit pointers.

This quantity is often referred to as the platform word size, though the term has several (conflicting) meanings which limit its usefulness.

Because pointers today are either 64 bits long, or less commonly, 32 bits long, depending on the platform, Swift defines an integer type Int which is the same size as a platform pointer, whatever that size is. This type also has an unsigned companion called UInt. Most of the Swift standard library is built around Int, and so you should prefer it over UInt and the sized integer types whenever the integer’s binary format does not matter to you.
